# 우리 프로그램 전체 TIMEOUT 문제 및 회피 엔진 개선 계획

## 1. 문제 원인 분석: 왜 우리 엔진에서 모든 연결이 TIMEOUT 인가?

추가해주신 테스트 결과를 분석한 결과, **우리가 수정한 엔진을 켰을 때 도리어 인터넷 전체(GitHub, STUN 포함)가 먹통이 되는 치명적인 버그**가 포함되어 있었습니다.

**버그의 핵심 (WinDivertHelperParsePacket의 IP 길이 검증 실패):**
* C++ `Diverter.cpp` 에서 DPI 우회를 위해 TLS ClientHello 패킷을 5바이트와 나머지 사이즈로 분할(Fragmentation)하여 복사했습니다.
* 복사본 `packet1`, `packet2`의 헤더를 파싱하기 위해 `WinDivertHelperParsePacket` 함수를 호출하는데, 이때 복사본의 `IP 헤더.Length` 값은 여전히 **원본의 긴 길이**를 가지고 있습니다.
* 반면, WinDivert 프레임워크는 내부 보안상 **"IP 헤더에 적힌 길이보다 실제 버퍼의 길이가 짧으면 손상된 패킷으로 간주하고 파싱을 실패(FALSE)"** 처리합니다.
* 파싱이 실패하면서 `ip1`, `ip2` 포인터가 `NULL`이 되어 이후 IP 길이가 전혀 수정되지 않았고, 비정상적인 길이 정보를 가진 패킷이 윈도우 네트워크 스택에 주입되면서 스택 레벨에서 패킷이 전부 버려지거나, 엔진 쓰레드가 오작동/크래시(Crash)되어 엔진 전체가 다운되었습니다. 
* 결과적으로 TCP 우회뿐만 아니라 정상적으로 작동해야 할 STUN 기능까지 엔진이 죽으면서 모조리 TIMEOUT 났던 것입니다.

## 2. 해결 및 수정 방향

WinDivertHelper에 의존하여 패킷을 재파싱하는 방식 대신, 이미 알고 있는 메모리 오프셋을 활용하여 직접 IP 길이를 먼저 수정하는 방식을 사용해야 합니다.

**구체적인 수정 방법:**

### `Diverter.cpp`
* `memcpy`로 패킷을 복사한 직후, `WinDivertHelperParsePacket`을 다시 호출하지 않습니다.
* 복사본(`packet1`, `packet2`) 메모리에서 직접 IP 해더 구조체와 TCP 헤더 구조체의 위치를 원본 포인터의 오프셋 덧셈으로 가져옵니다. 
  *예: `PWINDIVERT_IPHDR ip1 = (PWINDIVERT_IPHDR)packet1;`*
* 가져온 구조체 포인터를 통해 IP 헤더의 `Length` 와 TCP 헤더의 `SeqNum` 등을 알맞은 크기로 조작합니다.
* 데이터의 무결성이 확보된 상태에서 `WinDivertHelperCalcChecksums` 를 호출하여 안전하게 체크섬을 재계산하고 이 조작된 패킷을 네트워크로 흘려보냅니다(WinDivertSend).
* 원래 패킷의 흐름은 `return;`으로 취소(스킵)합니다.
